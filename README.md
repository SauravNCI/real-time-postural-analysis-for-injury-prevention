# Real-time postural analysis for injury prevention in resistance training

End-to-end pipeline: video/webcam -> MediaPipe pose landmarks -> biomechanical
feature engineering -> per-rep safe/unsafe classification -> live feedback
(both a desktop OpenCV window AND a real-time Django web app). Built to match
the MSc dissertation's Research Method (Section 3): four classifiers compared
(Logistic Regression, Random Forest, XGBoost, CNN+BiLSTM), stratified 5-fold
CV, held-out test evaluation, SHAP interpretability, McNemar's significance
testing.

## mediapipe version -- read this before installing anything

mediapipe releases from roughly late 2025 onward (`0.10.30`+) **removed the
legacy `mp.solutions.pose` API** entirely. The in-between versions that
still had it (`0.10.22`-`0.10.29`, and it turns out `0.10.21` too) have also
been pulled from PyPI, so there is no installable version left with that
API. `requirements.txt` just requires a plain `mediapipe` (latest, unpinned)
and this project uses the current **Tasks API** instead, via
`src/pose_backend.py` -- every script (`extract_real_dataset.py`,
`pose_extraction.py`, `live_app.py`, the Django app, `biomechanics.py`) goes
through that module, so you never touch mediapipe's API directly.

The Tasks API needs a small model file (`pose_landmarker_full.task`,
~30MB) that isn't bundled in the pip package. `pose_backend.py` downloads it
automatically to `models/pose_landmarker_full.task` the first time you run
anything that uses pose detection -- needs a normal internet connection. If
that download fails (restrictive network), download it yourself from the
URL in `pose_backend.py`'s docstring and save it to that exact path.

Also: use **Python 3.9-3.12** -- mediapipe has no wheel for 3.13+ yet.

## The core limitation: there is no expert safe/unsafe ground truth

Your real dataset (`data/<exercise>/*.mp4`) is exercise-type-labelled only
(folder name = "squat", "deadlift", etc.) -- like the Kaggle source it comes
from. **Nothing in the raw data says whether a rep's form was safe or
unsafe.** `extract_real_dataset.py` handles this by calling
`label_rules.py`'s automatic heuristic: a rep is flagged "unsafe" if it's a
statistical outlier (robust z-score, computed per-exercise) on trunk lean,
knee symmetry, or trunk stability -- the injury-relevant signals from the
dissertation's literature review. **Read `label_rules.py`'s docstring in
full** -- it explains exactly why this design was chosen and states plainly
that it is a placeholder for expert annotation, not a validated clinical
label. State this explicitly in your dissertation's limitations section;
see the wording suggested at the top of that file.

If you get even a small expert-labelled subset later (see
`data/raw/LABELLING_PROTOCOL.md`), pass it as `--labels_csv` to
`extract_real_dataset.py` -- real labels always override the heuristic for
the videos they cover.

## Synthetic-data pipeline validation (for reference)

Before real video was available, the full pipeline (everything except the
video/camera-dependent steps) was validated end-to-end on a synthetic
dataset (650 reps, realistic class overlap + ~8% label noise), to prove the
code itself is correct:

```
| Model                | Accuracy | Precision | Recall | F1    | Meets recall >=0.85 |
|-----------------------|----------|-----------|--------|-------|----------------------|
| logistic_regression   | 0.838    | 0.765     | 0.812  | 0.788 | No                   |
| random_forest         | 0.854    | 0.822     | 0.771  | 0.796 | No                   |
| xgboost               | 0.846    | 0.804     | 0.771  | 0.787 | No                   |
| cnn_bilstm            | 0.646    | 0.511     | 0.938  | 0.662 | Yes                  |
```

The CNN-BiLSTM was the only model clearing the recall >= 0.85 target, at a
real precision cost -- a legitimate precision/recall tradeoff, confirmed
significant by McNemar's test (p < 0.001 vs. all three other models). SHAP
correctly recovered `trunk_lean_angle` and `knee_symmetry` as top predictors,
confirming the interpretability pipeline works correctly. This table will be
superseded by your real results once you run the commands above --
regenerate it and swap it into your dissertation's results section.

## Design decisions worth noting in your dissertation write-up

- **Why the live paths (desktop + Django) use RF/XGBoost, not CNN-BiLSTM**:
  real-time inference needs milliseconds with no GPU dependency; tree models
  satisfy that trivially. CNN-BiLSTM stays in the offline comparison table.
- **Why SHAP only runs on the tree-based model**: matches the dissertation's
  stated plan, and `TreeExplainer` is exact and fast, unlike `DeepExplainer`
  on the LSTM.
- **Why `biomechanics.py` is a single shared module**: guarantees identical
  feature computation across extraction, training, `live_app.py`, AND the
  Django app -- avoids train/serve skew by construction. Worth stating
  explicitly in your methodology section.
- **Why `label_rules.py` exists at all**: the raw dataset has no safety
  label; this makes the labelling gap and its resolution fully transparent
  and auditable (`n_violations` / `label_source` columns) rather than
  silently guessing.
- Unit tests exist for the core angle/segmentation math
  (`test_biomechanics.py`, 5/5 passing) -- cite this as basic software
  quality assurance for a dissertation with a safety-relevant claim.

## Known limitations to state explicitly in the dissertation

1. **Labels are heuristic, not expert-verified** (see `label_rules.py`).
   This is the single most important limitation -- performance figures show
   the pipeline works correctly, not that the safety classification is
   clinically validated.
2. `trunk_lean_angle` (shoulder-hip-ankle angle) approximates lumbar/spinal
   alignment, not a direct measurement -- true lumbar lordosis needs
   marker-based mocap.
3. `RepSegmenter` thresholds are reasonable defaults, not calibrated to your
   specific camera angles -- the fallback in `extract_real_dataset.py`
   (treating a whole video as one rep when segmentation fails) partially
   mitigates this but expect some noisy rep boundaries on real footage.
4. The synthetic-data comparison table above is a pipeline sanity check, not
   a substitute for the real-data results once `train_models.py` is run
   against `rep_features_real.csv`.
